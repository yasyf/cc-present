# Troubleshooting

**Bare `cc-present` isn't found — or answers with an old version.** Bare invocation relies on Claude Code ≥ 2.1.91 putting each plugin's `bin/` on the Bash tool PATH, and that entry is version-pinned in the session's shell snapshot. A snapshot captured before the plugin was installed leaves the command not found; one captured before the last plugin update keeps resolving the superseded binary, so bare `cc-present` silently runs an older version than the plugin ships — compare `cc-present --version` when in doubt. Both cases have the same remedy: invoke the absolute path `"${CLAUDE_PLUGIN_ROOT}/bin/cc-present"`, which always names the current install, and start a fresh session to rebuild the snapshot. A missing binary file is the next entry.

**The binary is missing.** Run `bash "${CLAUDE_PLUGIN_ROOT}/scripts/install-binary.sh"`. It downloads the release binary for this platform into `bin/cc-present`. The SessionStart hook runs it automatically on session start, and `mcp-channel.sh` runs it again before exec'ing the channel server, so a missing binary usually means both paths failed — run it by hand and read its stderr.

**`watch` prints nothing and never exits.** You omitted `--session`. The artifact commands default it from `$CLAUDE_CODE_SESSION_ID`; `watch` takes only the flag, and with an empty session it polls for a subject forever. Relaunch it as `watch --session "$CLAUDE_CODE_SESSION_ID"`. Confirm the Monitor wrapping it was launched with `persistent: true`, and check the daemon with `status`.

**`start` says `child did not come up ... within 30s` right after an upgrade.** A daemon older than 0.11.0 speaks the pre-daemonkit control-socket framing, and the new daemon fails its incumbent health check (`wire: frame exceeds maximum`) instead of evicting it — the upgrade cannot self-heal across that boundary. Stop the old daemon once by hand — its own binary's `stop`, or `pkill -f 'cc-present.* daemon'` — then re-run `start`. One-time per machine crossing 0.10.x → 0.11.0.

**The daemon looks stale or version-skewed.** `status` prints the running daemon's version. Every artifact command self-heals skew: a strictly older daemon is evicted and replaced on the next invocation; a same-version rebuild ties and keeps the running daemon — `stop` to force the swap. To force it, `stop` — the next command lazy-respawns a fresh daemon. State survives: the sqlite log, cursors, and the artifact all persist across restarts.

**`channel: pending` or `inactive` printed.** The status line now reports honest channel presence. `active` appears when the MCP channel from the resolved Claude session is attached: a click streams straight back as a `<channel source="cc-present">` tag, so skip the Monitor. `pending` or `inactive` now signals a real absence, not an unproven placeholder — either the channel is not wired (this session's MCP channel server never attached) or the resolved Claude session is gone (its window process no longer lives). Check that the channel server is configured for this session and that you are still inside the Claude session that launched the artifact; while the channel is absent, arm the Monitor on `watch` to keep receiving interactions.

**The URL stopped working.** The daemon retries its previous port when it restarts (upgrade, `stop`, reboot), falling to a fresh one only when the old port is taken — so a stale URL usually means a port conflict happened. Re-run `start` — it resumes the same artifact and reprints the URL on the current port. Give the user the fresh URL verbatim; a health check is `curl -fsS http://127.0.0.1:<port>/api/health` expecting `{"version":…}` — never a bare status-code probe, since the SPA fallback answers unmatched paths with a 200 HTML shell.

**No `tailnet:` line printed.** The daemon publishes tailnet legs only under synckit mesh trust. Inspect with `cc-present trust`. No mesh state at daemon start: register the machine in synckit, then `stop` and re-run `start` (trust fixes at daemon construction). Tailscale down: legs appear within ~30s of `tailscale up` — re-run `start` to print them, no restart needed. The tailnet URL serves machines in the mesh; a non-mesh device gets 401 and needs the `pair` QR/token flow. Never work around a missing `tailnet:` line with `tailscale serve` or Funnel — proxied requests reach the daemon as loopback peers with a foreign Origin, which the auth model rejects (and would bypass trust if it didn't).

**`push` (or `start --doc`) was rejected.** Each validation error names its offending block id — a duplicate id, an unknown type, a missing required field, a bad `progress` range, an over-cap image or document. `push --dry-run` prints every violation at once, one per line, so fix them all in a single pass and re-run until it prints `ok`; a malformed JSON file reports the line and column instead. A rejected push changes nothing — the board keeps rendering the last good document.

**A dotted block type was rejected at push.** `pack block type "example.rating" is not installed` means no installed pack supplies that type. Run `pack list` and read where the pack landed. Absent entirely: discovery skipped it — a pack must be an installed plugin with a `.claude/components/cc-present.toml`, or listed in the config's `packDirs`. Under `dropped:` with a reason: fix what the reason names — a manifest violation, `duplicate pack name … in same tier`, `shadowed by dev dir`, or `disabled by config`. Dropped with `host_api N, want 1..2`: the pack requires a newer host API than this cc-present implements — upgrade cc-present. Once `pack list` shows the dotted type, `push --dry-run` revalidates offline. Authoring a new pack is the `cc-present:author-pack` skill's job.

**Browser clicks fail after close / HTTP 409.** `close` is terminal. The REST edge answers any later interaction with `409 presentation is closed`, and a click that raced the close lands in the log as a harmless no-op — recorded outcomes are unaffected. Agent writes are refused too (`artifact <slug> is closed`). There is no un-close: to present again, run `start` — it detects the closed artifact and creates a fresh one.

**`no cc-present artifact for this window`.** `push`, `update-block`, `reply`, and friends need an existing artifact in this Claude window — run `start`; it prints the `session:`/`url:`/`channel:` lines and the rest of the commands resolve from there, from any directory. Commands that still resolve by directory mean the running daemon predates window-keyed resolution (`status` shows v0.9.0 or older, which keyed artifacts by the `start` directory): run `stop`, then `start` fresh — boards made under the old keying are not resolvable by the new daemon.

**The Monitor stopped under an event flood, or replayed old events.** Monitors stop themselves under a high event rate — re-arm it; `watch` resumes from its cursor, so nothing is lost. Delivery is at-least-once, so a re-armed Monitor (or a channel/Monitor overlap) can replay frames you already handled: replays are harmless, since decisions, choices, and inputs are last-write-wins and feedback dedupes by its `id`.

**An image block failed to inline.** `image "<path>" not found` means the src is a local path the CLI can't read from your cwd — use an absolute path. `exceeds 5242880` means the file is past the 5 MiB asset cap — compress it, or host it and use an `https://` src.
