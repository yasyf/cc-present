# CLI cheatsheet

Invoke every command as bare `cc-present` — the plugin's `bin/` rides the Bash tool PATH; when the bare name doesn't resolve — or resolves to a stale version — fall back to `"${CLAUDE_PLUGIN_ROOT}/bin/cc-present"` (see `troubleshooting.md`). Every command is a thin call to the local daemon, which lazy-starts on first use. Document and block JSON always travels as a file path (or `-` for stdin), never inlined in the argument list. `--session` resolves the artifact for this Claude window; an artifact belongs to the window (session id + claude pid), never to a directory, so every command works from any cwd; `--cwd` is accepted and recorded but no longer affects resolution. The artifact commands default `--session` to `$CLAUDE_CODE_SESSION_ID`, but `watch`, `status`, and `channel-ack` do **not** — pass it explicitly.

**Who runs what**: with a board live, the event-reaction commands — `outcomes`, `reply`, `update-block`, `remove-block`, `revising` — belong to the dispatched `present-handler` agent; the main session authors (`start`, `push`, new-round upserts), steers (`direct`), and owns the round lifecycle (`round`, `close`).

| Command | What it does |
|---|---|
| `cc-present start --session <id> [--doc <file\|->] [--title <t>] [--new]` | Create or resume this window's artifact and print: `session: <subject-id>`, `url: http://127.0.0.1:<port>/p/<slug>--<hash>`, zero or more `tailnet: http://<magicdns-or-ip>:<port>/p/<slug>--<hash>` lines, `channel: active\|pending\|inactive` (`active`: channel proven — skip the Monitor; `pending`/`inactive`: arm the Monitor on `watch`). A `tailnet:` line is the URL to hand the user for viewing from another machine — it typically carries a **different port** than `url:`, and it works token-free only from machines in the synckit mesh (trust is source-IP; a non-mesh device such as a phone's browser gets 401 and needs the `pair` QR/token flow). Never put the board behind `tailscale serve`/Funnel — the contract bans it (proxied requests defeat the auth model). `--doc` seeds the document (validated first); `--title` names the URL slug when no `--doc` is given; `--new` forces a fresh artifact. A resume that lands on a closed artifact goes fresh automatically. |
| `cc-present push <file\|-> [--round <current\|new>] [--round-title <t>] [--dry-run]` | Replace the whole document; prints `revision: <n>` first (parse that line), then `url:` and any `tailnet:` lines like `start`, plus `round: <n>` when the push opened a new round. A push that introduces a new top-level block into a mid-review round — dirty, and the human has already interacted — is rejected unless `--round` declares intent: `current` joins the round in progress, `new` closes it into history and opens the next (`--round-title` names it; valid only with `--round new`). `--dry-run` validates offline (no daemon, no artifact needed): `ok` on pass, otherwise every violation one per line — each naming its offending block id — and a non-zero exit. Malformed JSON fails with `(line N, column M)` appended to the decode error. |
| `cc-present update-block <file\|-> [--after <id>] [--round <current\|new>] [--round-title <t>] [--dry-run]` | Insert or replace a single block (the file holds one block's JSON). A NEW top-level id landing in a mid-review round (dirty + human-interacted) is rejected unless `--round` declares intent — `current` extends the round, `new` advances it with the unanswered blocks carried forward (prints `round: <n>`; `--round-title` names it); re-upserts of existing ids never need the flag. An existing id is replaced in place, wholesale, wherever it lives — top level, or a card child inside its card. A new id inserts after `--after`: a top-level `--after` inserts at top level, a card-child `--after` inserts into that card after the named child; omitted `--after` appends at top level, and an unknown or option-visual `--after` errors (`omit --after to append at the end`). Silent on success. `--dry-run` validates the lone block offline, printing every violation like `push --dry-run`; the block is checked in a synthetic single-block envelope, so only block-level violations surface, never envelope ones. |
| `cc-present remove-block <id>` | Remove a block by id — a top-level block, or a card child (spliced from its card, which restamps into the current round). An unknown id errors (`no block %q in the current document`); an option-visual id errors pointing at its choice. Silent on success. |
| `cc-present reply --block <id> --body <markdown>` | Append an agent reply to a block's thread; renders in the browser immediately. Returns at once; silent on success. |
| `cc-present round [--title <t>]` | Advance or title the round; prints `round: <n>` (the resulting current round). Right after a submit the board is clean, so it only titles the round the submit already opened; mid-round (you've upserted blocks since the last boundary) it snapshots the current round into history first, then advances and titles the new one — carrying every unanswered block forward, so advancing never freezes the human's open controls. Clears any revising announcement. Never bumps the revision. |
| `cc-present revising [blockId...] [--note <text>] [--clear]` | Announce the block ids you're rewriting, with an optional `--note` shown on each. A card-child id resolves to its enclosing card (the card wears the badge); an unknown or option-visual id errors. Warn-only — the human's controls stay live; it badges those steps as in-flight. An `update-block` or `remove-block` clears a block's mark (completion is implicit), so the path is announce-once then upsert — but a rewrite that lands under a **new** id clears nothing on its own: follow it with `revising --clear`, or remove the superseded block. A submit or `round` boundary drops the whole set too. A bare `revising` with no ids — or `--clear`, which refuses ids and `--note` — abandons the announcement; no ids with `--note` is the doc-level "drafting a new step" state. Silent on success. |
| `cc-present outcomes [--no-doc] [--block <id>]` | Print the reduced state as indented JSON: the current document plus `interactions` (decisions, choices, inputs, feedback, replies, submitted, closed) and `rounds` (current round, per-block round stamps, closed-round history). The post-submit drain, run by the handler. `--no-doc` omits the document, printing only the interactions and rounds. `--block <id>` filters to one block — its interactions only, and (with the doc included) just its subtree; a child id keeps its own entries plus the enclosing card; an unknown id errors. |
| `cc-present close [--summary <text>]` | Terminally close the artifact, recording `--summary` on the `present.closed` event; prints `closed: <slug>`. After close, browser interactions get 409 and agent writes are refused. |
| `cc-present watch --session <id> [--once]` | Print one JSON event per line, each payload self-describing via its embedded `type`, and exit on the terminal `present.closed`. It delivers human events, presence frames, and that terminal close; your own agent writes never appear. Resumes from a persisted cursor, so a restart re-delivers nothing already emitted. `--once` exits after the first event instead. `--session` is required in practice: without it, `watch` polls for a subject forever. Degraded mode only — with the handler dispatched, nothing should consume this feed (see `troubleshooting.md`). |
| `cc-present status [--session <id>]` | Show the daemon version, its HTTP address, and this window's subject (`open`/`closed`, or `none`). For every subject the daemon knows, use `sessions`. |
| `cc-present sessions` | List every subject the daemon knows for this scope — subject id, slug, bound session, open/closed status, event count, URL — under a `cc-present daemon <version> · port <n>` header. The "where did my board go" answer; never spelunk the sqlite db. |
| `cc-present direct <text> [--agent <id>]` | Drop operator guidance into a running handler agent's mailbox; it arrives at its next park or tool call. Defaults to the sole running handler; with none or several, errors naming them. Never sends to the top-level mailbox implicitly. |
| `cc-present version` | Print the client build version (same output as `--version`). |
| `cc-present stop` | Stop the background daemon (it lazy-respawns on the next command). |
| `cc-present channel-ack --session <id>` | Mark this window's channel proven after the first delivered `<channel>` tag, flipping later `start`s to `channel: active`. Run it once when a tag arrives while the Monitor is armed, then stop the Monitor. |
| `cc-present pack list` | List the installed block packs, offline (no daemon). Per pack: name and version, its directory, the absolute path of its reference fragment, and each block's dotted type with an `(interactive)` marker; `dropped:` lists every skipped candidate with its reason. Prints `no packs installed` when none. |
| `cc-present pack lint <dir>` | Validate a pack root fail-loud, offline: strict manifest, exact `host_api = 1`, declared files present, every schema compiling, and every declared example validating against its block schema. Prints `ok: <name> <version> (N blocks)`, or the first violation and a non-zero exit. |
| `cc-present pack init [--name <n>] <dir>` | Scaffold a new block pack into `<dir>` from the embedded reference pack, offline, renamed to `--name` (defaults to the dir's basename). Writes 23 files including a `.gitignore`; refuses a non-empty dir. Prints the file count, the next steps (`bun install`, `bun run build` — `pack lint` needs the built `dist/pack.js` — `bun run smoke`, `pack lint .`), and the absolute path to add to `packDirs`. |

`image` blocks whose `src` is a local file path are inlined by `start --doc`, `push`, and `update-block`: the file is read (5 MiB cap), stored content-addressed on the daemon, and the src rewritten to its `asset:<sha256>` ref. `push --dry-run` performs the same read locally, so a missing or oversized image fails the dry run too.

The remaining subcommands — `daemon`, `channel`, `session-record`, `completion`, and the agent-plane quartet `agent-start`/`agent-inject`/`agent-stop`/`agent-report` — are plumbing the plugin's hooks and MCP config invoke; never run them from the skill flow.

The daemon keeps state under `~/.cc-present` (sqlite db, control socket, HTTP handshake, per-consumer cursors).
